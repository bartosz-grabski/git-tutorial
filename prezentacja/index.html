Git

Git

Acceptance criteria Linusa Torvaldsa:

  • Wziąć przykład z CVS, czego nie robić.
  • System powinien być rozproszony.
  • System powinien być chroniony przed błędami w repozytorium (przypadkowymi, jak awaria twardego dysku, jak i złośliwymi, wprowadzonymi przez kogoś).
  • System powinien być szybki.

Git

Zalety Gita

  • Popularność (GitHub!)
  • Brak centralizacji
  • Branche, brache, branche
  • Możliwość szybkiego przełączenia kontekstu pracy
  • Prosty rozwój wielu linii produkcyjnych oprogramowania
  • Feature based workflow
  • Lokalne commity, dowolność w eksperymentowaniu nad kodem

Warsztat

Praca w parze - polecana

  • Kod synchronizujemy przez repozytorium na GitHubie, które stworzy jedna z osób w parze
  • Wybieramy sobie role z przedziału [1;2]
  • W zależności od roli interesować nas będzie lewa lub prawa kolumna poleceń, które będę przedstawiał.

Warsztat

Praca w pojedynkę

  • Symulujemy pracę w dwie osoby
  • Dwa różne repozytoria na dysku synchronizowane przez GitHuba
  • Będzie trzeba szybko klepać :)

Warsztat

Skrypt mayday

  • Umożliwia szybkie przywrócenie repozytorium do poprzedniego stanu jeśli coś zepsuliśmy
  • Umożliwia szybkie przeskoczenie krok dalej jeśli zabrakło nam czasu
  • Stany są wyraźnie oznaczone na stronie z listą poleceń
  • Uwaga! Skrypt modyfikuje repozytorium na GitHubie. Jeśli jedna osoba przywróci repozytorium do określonego stanu, druga też to musi zrobić. Osobie pracującej w pojedynkę wystarczy uruchomienie polecenia tylko raz.

Część II

Podsumowanie

Kilka dobrych rad:

  • Commituj często
  • Synchronizuj często głównego brancha
  • Trzymaj się na początku z daleka od git rebase
  • Nigdy, przenigdy nie używaj poleceń modyfikujących historię repozytorium na elementach istniejących już w centralnym repozytorium!
  • Nie wrzucaj do repozytorium niekompilującego się kodu

Podsumowanie

Kilka dobrych rad c.d.:

  • Naprawiaj wszystkie konflikty, nigdy nie wrzucaj ich do repozytorium
  • Wykorzystaj GUI lub IDE przynajmniej do commitowania
  • Stwórz własne aliasy dla poleceń lub użyj gotowych nakładek
  • Unikaj binarek (blobów) w repozytorium i zadbaj o porządny .gitignore
